## **1\. Introduction**

This document defines the **APTITUDE Presentation Profile** for interoperable credential verification across APTITUDE pilots.

It specifies a baseline, cross-pilot profile for the validation of trust artifacts aligned with the [APTITUDE Trust Specifications](https://aptitude-consortium.github.io/wp2-trust-specifications/latest/), the [references](https://aptitude-consortium.github.io/wp2-trust-specifications/latest/deliverable-2.1-trust/#references_3) therein, and the ARF.

The purpose of this RFC is to establish a stable starting point for interoperability across APTITUDE pilots by defining:

- impacted roles and components;
- normative requirements for entities involved in the validation of WRPAC and WRPRC certificates;
- interface definitions for the relevant protocols (OCSP, CRL, Status List Token);
- a conformance check catalogue that maps the normative requirements into verifiable checks;
- a test case matrix that maps the conformance checks into executable test cases for the **ITB+ conformance test cases**.

This RFC is intended to support interoperability first. Pilot-specific or sector-specific extensions may be added later, but they should not weaken the baseline requirements defined here.

---

## **2\. Scope**

This RFC profiles presentation under three normative tracks:

- **OCSP track** -- profiled under the validation of WRPAC status defined by ETSI 119 411-8 and RFC 6960;
- **CRL track** -- profiled under the validation of WRPAC status defined by ETSI 119 411-8 and RFC 5280;
- **TSL track** -- profiled under the validation of WRPAC status defined by ETSI 119 475 and Token Status List [IETF draft](https://datatracker.ietf.org/doc/draft-ietf-oauth-status-list/).

This RFC includes baseline requirements for:

- generation of OCSP Requests;
- generation of OCSP Response and their verification;
- CRL parsing and verification;
- Status List Token parsing and verification.

This RFC does **not** define:

- sector-specific certificate semantics (already defined in [WRPAC profile](https://aptitude-consortium.github.io/wp2-trust-specifications/latest/deliverable-2.1-trust/#wallet-relying-party-access-certificate) and [WRPRC profile](https://aptitude-consortium.github.io/wp2-trust-specifications/latest/deliverable-2.1-trust/#wallet-relying-party-registration-certificate));
- the lifecycle of the entities and related Trust Artifacts (already defined in [Trust Management](https://aptitude-consortium.github.io/wp2-trust-specifications/pr-46/pr-workspace/trust-management-process/));
- revocation artifacts and semantics (already defined in [Revocation Menchanisms](https://aptitude-consortium.github.io/wp2-trust-specifications/latest/deliverable-2.1-trust/#revocation-mechanisms));
- pilot-specific UI requirements;
- detailed test assertions; (these are derived into the *ITB+* test suite from the requirements matrix in this document).

## **4\. Roles and Components**

### **4.1 Wallet Relying Party**

The **<roles:Wallet Relying Party>**, as defined in the APTITUDE glossary and aligned with the ARF notions of Relying Party, Attestation Provider and PID Provider. In this RFC, it is the entity to whom the WRPAC and WRPRC refers.  

### **4.2 Wallet Unit**

The **Wallet Unit** is the <components:Wallet Unit> as defined in the ARF and APTITUDE glossary, including one or more <components:Wallet Instance|Wallet Instances> on the Holder’s device or environment. In this RFC, the term **Wallet** refers to the Wallet Unit (and its Wallet Instances) when the distinction is not material, acting on behalf of the <roles:Holder> to verify the status of the WRPAC and WRPRC received when transacting with a WRP.

### **4.3 Provider of WRPAC**

The **<roles:Provider of WRPAC>**, as defined in the APTITUDE glossary and aligned with the ARF notion of Provider of WRPAC, is the entity issuing WRPAC and the related status Services (CRL endpoint and/or OCSP responder).

### **4.4 Provider of WRPRC**

The **<roles:Provider of WRPRC>**, as defined in the APTITUDE glossary and aligned with the ARF notion of Provider of WRPRC, is the entity issuing WRPRC and the related status Services (Status List Token endpoint).

## **6\. Normative Requirements**

This section defines the baseline normative requirements for APTITUDE implementations of the issuance profile.

Unless otherwise stated, the requirements in this section apply to issuance interactions only.

### **6.1 Wallet Requirements**

A Wallet conforming to this RFC SHALL:

- Support RFC 5280 for CRL parsing and validation;
- Support Token Status List for Staus List Token Parsing and validation;

A Wallet conforming to this RFC SHOULD:

- Support RFC 6960 for OCSP request generation as defined in [OCSP request](https://aptitude-consortium.github.io/wp2-trust-specifications/latest/deliverable-2.1-trust/#online-certificate-status-protocol-request-format);
- Support RFC 6960 for OCSP response parsing and validation.

### **6.2 Provider of WRPAC Requirements**

A WRPAC Provider conforming to this RFC SHALL:

- expose at least a CRL endpoint as described in section [CRL Endpoint](#71-crl-endpoint);
- generate the CRL listing all revoked WRPAC with the semantics defined in [CRL](https://aptitude-consortium.github.io/wp2-trust-specifications/latest/deliverable-2.1-trust/#certificate-revocation-lists).

A WRPAC Provider conforming to this RFC MAY:
- expose the OCSP endpoint as described in section [OCSP Endpoint](#72-ocsp-endpoint) with OCSP responses conforming to [OCSP response](https://aptitude-consortium.github.io/wp2-trust-specifications/latest/deliverable-2.1-trust/#online-certificate-status-protocol-response-format).

### **6.3 Provider of WRPRC Requirements**

A WRPRC Provider conforming to this RFC SHALL:

- expose a Status List Token endpoint as described in section [SLT Endpoint](#73-tsl-endpoint);
- generate the status List, listing the status of all issued WRPRC, and embedding it into a Status List Token as defined in [TSL](https://aptitude-consortium.github.io/wp2-trust-specifications/latest/deliverable-2.1-trust/#token-status-list).

## **7\. Interface Definitions**

This section defines the logical interfaces for conformance testing. The following interfaces are added on top of these added in RFC001, RFC002.

### **7.1 CRL Endpoint**

- **Direction**: Wallet to Provider of WRPAC's CRL endpoint.
- **Method**: `GET`

#### **Request**

The Wallet SHALL construct an HTTP `GET` request to retrieve the CRL associated with the certificate.

- **URI Construction**:

  - The request URI is extracted directly from the `cRLDistributionPoints` extension of the certificate.
  - No additional query parameters are required unless explicitly specified by the URI in the CDP extension.

- **Headers**:

  - `Host`: SHALL contain the hostname from the CLR URI.
  - `Accept`: SHALL contain the value `application/pkix-crl`.

#### **Response**

The Provider's CRL endpoint SHALL respond with the appropriate HTTP status codes and the correctly formatted CRL file if the request is successful.

- **Status Codes**:

  - `200 OK`: The request was successful, and the CRL is included in the response body.
  - `4xx` / `5xx`: Standard HTTP error codes (e.g., `404 Not Found`, `500 Internal Server Error`) SHALL be used only if the server cannot process the HTTP request at the transport level.

- **Headers**:

  - `Content-Type`: SHALL be set to `application/pkix-crl` for successful responses containing the CRL.
  - `Content-Length`: SHOULD be included to indicate the size of the CRL file in bytes.

- **Body**:

  - The response body SHALL contain the CRL in ASN.1 DER format as specified in RFC 5270.

### **7.2 OCSP Endpoint**

- **Direction**: Wallet to Provider of WRPAC's OCSP endpoint.
- **Method**: `POST`

[OPTIONAL] If the DER encoding of the request is less than 255 bytes, the Wallet MAY use the `GET` method as defined in RFC 6960, by encoding the OCSP request in the URL. In this case, the interface is defined as follows:

- **Direction**: Wallet to Provider of WRPAC
- **Method**: `GET`

#### **Request**

As detailed in [OCSP request](https://aptitude-consortium.github.io/wp2-trust-specifications/latest/deliverable-2.1-trust/#online-certificate-status-protocol-request-format), the OCSP request is a DER encoded `OCSPRequest` structure, with the following parameters:

- `version`;
- `requestList` with a list of `Request` structures, each containing:

  - `reqCert` with the `CertID` of the certificate to be checked;

- `requestExtensions` with the `id-pkix-ocsp-nonce` extension containing a fresh nonce value.

The Wallet SHALL construct an HTTP `POST` request to the URI extracted from the Authority Information Access extension of the certificate, specifically the `accessLocation` associated with the `id-ad-ocsp` access method.

- **POST Headers**: The request SHALL include a `Content-Type` header set to `application/ocsp-request` and an `Accept` header set to `application/ocsp-response`.
- **POST Body**: The body of the `POST` request SHALL contain the binary DER-encoded `OCSPRequest` structure.
- **GET Construction (Optional)**: If utilizing the optional `GET` method, the request URI SHALL be constructed by appending a URL-encoded Base64 representation of the DER-encoded `OCSPRequest` to the base OCSP URI (e.g., {OCSP_URI}/{base64_encoded_request}). The GET request SHALL include an `Accept` header set to `application/ocsp-response` and has no message body.

#### **Response**

As detailed in [OCSP response](https://aptitude-consortium.github.io/wp2-trust-specifications/latest/deliverable-2.1-trust/#online-certificate-status-protocol-response-format), the OCSP response is a DER encoded `OCSPResponse` structure, with the following parameters:

- `responseStatus` with the status of the response;
- `responseBytes` with the `responseType` and the `response` containing the DER encoded `BasicOCSPResponse` structure, which includes:

  - `tbsResponseData` with the list of `SingleResponse` structures, each containing:

    - `certID` matching the request;
    - `certStatus` with the status of the certificate (`good`, `revoked`, or `unknown`);
    - `thisUpdate` and optionally `nextUpdate` for temporal validity checks.

  - `signatureAlgorithm` and `signatureValue` for cryptographic validation;
  - `responseExtensions` with the `id-pkix-ocsp-nonce` extension containing the same nonce value as in the request.

If the request is successful, the Provider's OCSP endpoint SHALL respond with the appropriate HTTP status codes and the correctly formatted OCSP response:

- **Status Codes**:

  - `200 OK`: The request was successfully received and processed. Note that a 200 OK HTTP status SHALL be returned even if the internal OCSP `responseStatus` indicates an error (e.g., `malformedRequest`, `internalError`, `tryLater`), provided the HTTP transport layer itself succeeded.
  - `4xx` / `5xx`: Standard HTTP error codes (e.g., `404 Not Found`, `500 Internal Server Error`) SHALL be used only if the server cannot process the HTTP request at the transport level.

- **Headers**:

  - `Content-Type`: SHALL be set to `application/ocsp-response`.
  - `Content-Length`: SHALL accurately reflect the size of the DER-encoded OCSP response in bytes.
  - `Cache-Control` / `Expires`: MAY be included to govern HTTP caching behaviors, in case it SHALL align with the `nextUpdate` field provided in the internal payload.

- **Body Format**:

  - The HTTP response body SHALL contain the binary DER-encoded `OCSPResponse` structure.

### **7.3 TSL Endpoint**

- **Direction**: Wallet to Provider of WRPRC  
- **Method**: `GET`

#### **Request**

The Wallet SHALL construct an HTTP `GET` request to retrieve the Token Status List JWT.

- **URI Construction**:

  - The request URI SHALL be extracted from the `uri` value of the `status_list` claim found within the token or credential being verified.

- **Headers**:

  - `Host`: SHALL contain the hostname extracted from the Status List URI.
  - `Accept`: SHOULD include `application/statuslist+jwt` to explicitly indicate the Wallet's capability and expectation to receive a Status List JWT.

#### **Response**

The Provider's Status List endpoint SHALL respond with the appropriate HTTP status codes and the correctly formatted Token Status List JWT.

- **Status Codes**:

  - `200 OK`: The request was successful, and the Status List JWT is included in the response body.
  - `4xx` / `5xx`: Standard HTTP error codes (e.g., `404 Not Found`, `500 Internal Server Error`) SHALL be used only if the server cannot process the HTTP request at the transport level.

- **Headers**:

  - `Content-Type`: SHALL be set to `application/statuslist+jwt`.

- **Body**:

  - The response body SHALL consist of a Token Status List JWT.
  - The payload of this JWT contains claims as defined in [Status List Token](https://aptitude-consortium.github.io/wp2-trust-specifications/latest/deliverable-2.1-trust/#status-list-token).

## **8\. Privacy and Security Considerations**

### **8.1 Privacy Considerations**

While the WRPAC and WRPRC certificates identify organizational entities (Attestation/PID Providers and Relying Parties) and do not contain Personally Identifiable Information (PII), however, the act of validating these certificates has significant privacy implications for the end-user operating the Wallet.

- When a Wallet executes a live OCSP request for a specific Relying Party's certificate, the serial number of that certificate is transmitted over the network. A network observer can correlate the OCSP request with the Relying Party, revealing the user's interactions.
- To obscure the specific endpoints being queried and the parameters of the request, Wallets SHOULD mandate the use of HTTPS (TLS) for all CRL, OCSP, and TSL network queries, despite the payloads already being cryptographically signed.
- Implementers concerned with high-privacy environments SHOULD favor the use of Status List Tokens (TSL) or cached CRLs over real-time OCSP queries. By downloading a batched list of statuses, the Wallet obfuscates the specific credential or entity it is actively verifying, blending the user's specific query into a larger cryptographic dataset.

### **8.2 Security Considerations**

Implementations of this specification SHALL account for multiple attack vectors against the PKI and revocation infrastructure. The following considerations establish the baseline security posture for Wallets and Providers of certificates.

#### **8.1.1 Cryptographic Integrity and Trust Anchors**

Implementations SHALL NOT rely on deprecated cryptographic algorithms (e.g., SHA-1, RSA-1024) for the signature generation or validation of CRLs, OCSP responses, or Status List Tokens (SLTs).

Wallets SHALL thoroughly validate the certification path of the OCSP responder, CRL issuer, or TSL issuer up to a trusted root, verifying that the signer is explicitly authorized to issue revocation data for the target certificate (e.g., checking the `cRLSign` bit or `id-kp-OCSPSigning` Extended Key Usage).

#### **8.1.2 Freshness and Replay Attacks**

To prevent attackers from capturing a valid `good` OCSP response and replaying it after a certificate has been revoked, Wallets SHALL include the `id-pkix-ocsp-nonce` in requests and strictly enforce its matching presence in the response.

Wallets SHALL strictly validate the `thisUpdate`, `nextUpdate`, `iat`, and `exp` fields across all tracks. Systems SHOULD account for minor acceptable clock skew between the Provider and the Wallet, but SHALL reject responses where temporal validity has unambiguously expired, as these may indicate a downgrade or replay attack.

#### **8.1.3 Denial of Service (DoS)**

Providers or WRPAC and WRPRC SHALL ensure high availability of CRL, OCSP, and TSL endpoints. Wallets SHALL define a clear policy on how to behave if the revocation endpoint is unreachable (i.e., whether to trust the entity, or abort the transaction).

#### **8.1.4 Transport Security**

While the integrity of CRLs, OCSP responses, and TSLs is guaranteed by their digital signatures rather than the transport layer, the use of plain HTTP exposes the connection to manipulation. Attackers could intercept requests and return TCP RST packets or spoof 404 Not Found or 500 Internal Server Error responses, forcing the Wallet into a fail-open or fail-closed state without ever evaluating the cryptographic payload.

## **9\. Conformance**

An implementation **conforms to this specification as a Wallet** if it:

1. implements the requirements defined in [Section 6.1](#61-wallet-requirements);
2. supports the interfaces defined in [Section 7](#7-interface-definitions) for all the relevant tracks (CRL, OCSP, TSL);
3. successfully passes all the conformance checks defined in the [Conformance Check Catalogue](#101-conformance-check-catalogue) for the relevant tracks.

An implementation **conforms to this specification as a Provider of WRPAC** if it:

1. implements the requirements defined in [Section 6.2](#62-provider-of-wrpac-requirements);
2. supports the interfaces defined in [Section 7](#7-interface-definitions) for the CRL and OCSP tracks;

An implementation **conforms to this specification as a Provider of WRPRC** if it:

1. implements the requirements defined in [Section 6.3](#63-provider-of-wrprc-requirements);
2. supports the interfaces defined in [Section 7](#7-interface-definitions) for the TSL track;

### **10.2 Deployed Test Case Matrix**

The test matrix codes are generated by the three tracks defined in this RFC (CRL, OCSP, TSL), the test case ID is defined as follows:

- An initial code describing the track (`CRL`, `OCSP`, `TSL`).
- A second code for the type of checks (`REQ` for request schema, `RES` for response schema, `VAL` for validation checks).
- An increasing natural number.

| Test Case ID | Deployed Scenario | Request Profile / Variant | Checks Covered | Expected Outcome |
| :--- | :--- | :--- | :--- | :--- |
| **CRL-REQ-01** | Standard CRL Retrieval | HTTP GET via CDP | Wallet sends GET request with valid `Host` and `Accept: application/pkix-crl` headers. | Response returns `application/pkix-crl` and file is valid ASN.1 DER. |
| **CRL-RES-01** | Valid CRL Response | Provider Successful Serve | Provider returns `200 OK`, valid `Content-Type`, and binary DER CRL file. | Successful download of the CRL payload. |
| **CRL-VAL-01** | Freshness Verification | Temporal Check | Comparison of `current_time` against `thisUpdate` and `nextUpdate` fields. | CRL is accepted if `thisUpdate` ≤ `current_time` < `nextUpdate`; otherwise, attempt refresh. |
| **CRL-VAL-02** | Issuer Identification | Direct CRL Matching | Comparison of CRL issuer field with the certificate's issuer field (when DP `cRLIssuer` is omitted). | Binary match between Name structures; proceed if identical, otherwise reject CRL. |
| **CRL-VAL-03** | Indirect CRL Binding | `cRLIssuer` Present | Match CRL issuer field with `cRLIssuer` in DP; verify `indirectCRL` boolean in IDP is `TRUE`. | Match successful; CRL is used as an authoritative source for non-issuer revocations. |
| **CRL-VAL-04** | IDP Scope: Distribution | Partitioned CRL | Match `distributionPoint` name in CRL IDP extension with the name in the certificate's DP. | Exact string match on the URI or `Distinguished Name`; CRL accepted for that scope. |
| **CRL-VAL-05** | IDP Scope: Cert Type | EE vs. CA Scope | Verify `onlyContainsUserCerts` or `onlyContainsCACerts` against certificate `basicConstraints`. | Rejection of the CRL if an EE cert is checked against a CA-only CRL (and vice-versa). |
| **CRL-VAL-06** | Path Integrity | Signer Authorization | Validation of the CRL issuer's certification path to the same trust anchor as the target certificate. | Successful path validation; verification that the CRL signer's certificate has the `cRLSign` bit set. |
| **CRL-VAL-07** | Signature Verification | Cryptographic Check | Validation of the digital signature on the complete CRL using the validated public key of the CRL issuer. | Signature is cryptographically valid; CRL data is considered authentic. |
| **CRL-VAL-08** | Status Resolution | Revocation Search | Search for the certificate's serial number within the `revokedCertificates` list. | If found, `cert_status` matches the reason code; if not found, status is `VALID`. |
| **OCSP-REQ-01** | Standard OCSP Status Request | HTTP POST | Wallet sends POST to AIA URI with valid OCSP MIME types and DER body. | `200 OK` with a valid binary `OCSPResponse` payload. |
| **OCSP-REQ-02** | Short OCSP Status Request | HTTP GET (Optional) | Wallet successfully encodes < 255 byte request as Base64 in URL parameter. | `200 OK` and successful HTTP caching where applicable. |
| **OCSP-RES-01** | Valid OCSP Response | Provider Successful Serve | Provider returns `200 OK` (transport level success), valid MIME type, and DER response. | Successful download of the OCSP payload. |
| **OCSP-VAL-01** | Response Type Check | Basic Response Type | Verification that `responseType` OID matches `id-pkix-ocsp-basic` (`{1 3 6 1 5 5 7 48 1 1}`). | Reject if the response is not the basic type; OCSP clients SHALL support this type. |
| **OCSP-VAL-02** | Cryptographic Integrity | Digital Signature | Verification of the signature value across the hash of the DER encoded `ResponseData`. | Signature SHALL be valid; reject response if cryptographic check fails. |
| **OCSP-VAL-03** | Responder Authorization | CA Signature | Check if the response is signed by the CA that issued the certificate in question. | Proceed to status evaluation if the CA signature is verified. |
| **OCSP-VAL-04** | Responder Authorization | Authorized Responder | Verification of `id-kp-OCSPSigning` (OID `{1.3.6.1.5.5.7.3.9}`) in the Extended Key Usage extension of a responder certificate issued directly by the CA. | Proceed if Extended Key Usage is present and the signer matches the intended recipient. |
| **OCSP-VAL-05** | `CertID` Correlation | Request-Response Binding | Match the `CertID` (`hashAlgorithm`, `hashes`, `serialNumber`) in the response to the original request. | Reject if the response does not correspond to the requested certificate. |
| **OCSP-VAL-06** | Temporal Validity | Freshness Check | Comparison of `thisUpdate` against local system time to ensure it is "sufficiently recent". | Reject if `thisUpdate` is in the future or considered stale by local policy. |
| **OCSP-VAL-07** | Temporal Validity | `nextUpdate` Check | Verification that current time is less than `nextUpdate` (if provided). | Reject as unreliable if system time is past `nextUpdate`. |
| **OCSP-VAL-08** | Status Resolution | Final Status Mapping | Retrieval of `certStatus`: `good`, `revoked` (with reason/time), or `unknown`. | Target certificate is accepted only if status is `good`. |
| **OCSP-EXT-01** | Replay Protection | Nonce Validation | Verification that the `nonce` in the response matches the `nonce` sent in the `requestExtensions`. | Reject response if nonces do not match, preventing replay attacks. |
| **TSL-REQ-01** | Standard TSL Retrieval | HTTP GET via `status_list` | Wallet extracts URI correctly and sends GET request with required `Host` header. | Status Provider returns 2xx and the raw Status List Token. |
| **TSL-RES-01** | Valid TSL Response | Provider Successful Serve | Provider returns `200 OK`, `Content-Type: application/statuslist+jwt`, and JWT payload. | Successful resolution of the SLT from the endpoint. |
| **TSL-VAL-01** | Referenced Token Pre-check | Referenced Token Validation | Verification of signature, attributes, and `exp` claim of the token before checking status. | Proceed to SLT fetching only if Referenced Token is valid; otherwise reject. |
| **TSL-VAL-02** | Status Claim Syntax | JOSE/COSE Reference | Check for existence of `status` and `status_list` claims in the Referenced Token; validate `idx` and `uri` syntax. | Validation succeeds if structure adheres to JOSE/COSE rules. |
| **TSL-VAL-03** | SLT Integrity | Cryptographic Validation | Signature verification of the SLT (JWT or CWT format) using validated keys. | Reject SLT if signature is invalid or undetermined. |
| **TSL-VAL-04** | Subject Binding | URI Consistency | `sub` (JWT) or `2` (CWT) claim in SLT SHALL exactly match the `uri` claim in the Referenced Token. | Reject SLT if identifiers are not equal. |
| **TSL-VAL-05** | Temporal Validity | Expiration & Freshness | Verification of `iat`, `exp`, and `ttl` claims; comparison against `current_time`. | Reject SLT if current time is past exp. |
| **TSL-VAL-06** | Parsing & Decompression | SLT Content Extraction | Decompression of the `lst` claim using DEFLATE/ZLIB. | Successful recovery of the uncompressed bit array. |
| **TSL-VAL-07** | Index Boundary Check | Bit Array Access | Verification that the `idx` from the Referenced Token is within the bounds of the decompressed bit array. | Reject Referenced Token if index is out of bounds. |
| **TSL-VAL-08** | Status Resolution | Status Type Evaluation | Retrieval of bits at `idx` and mapping to Status Type (e.g., `0x00`=`VALID`, `0x01`=`REVOKED`). | Final decision made based on the bit value at the specific index. |
